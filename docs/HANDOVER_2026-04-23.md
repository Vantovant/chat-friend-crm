## Executive Summary

Today's work delivered two major upgrades to Vanto CRM:

1. **WhatsApp AI Trainer Layer (v6.2)** — A no-code admin surface that lets the operator teach, correct, and override the WhatsApp auto-reply AI without editing source code. Trainer rules sit on top of the Knowledge Vault and are injected into the system prompt at reply time, with three priority tiers (`advisory`, `strong`, `override`).
2. **Twilio / Maytapi Privacy Lockdown** — The administrator's WhatsApp / Twilio numbers and sensitive integration settings are now hidden from non-admin users at the database (RLS), edge function (role check), and UI layers.

Both changes are live in production.

## 1. WhatsApp AI Trainer Layer

### Problem Solved
The auto-reply AI was technically grounded in the Knowledge Vault but kept making practical business mistakes — most notably recommending `PWR` as a single product when the real catalog distinguishes:
- **PWR Lemon** → for men
- **PWR Apricot** → for women

Previously, fixing this required editing the edge function source. There was no operator-facing surface to teach the AI new business rules.

### Solution Implemented

A **three-layer correction system** added on top of the existing v6.1 hybrid retrieval (chunk search → raw_text fallback → honest fallback):

```
Inbound WA message
   ↓
1. Knowledge Vault grounding (unchanged)
   ↓
2. Trainer rule matching (NEW)
   ↓
3. Final answer generation (now respects trainer overrides)
```

### Database Schema

**New table: `public.ai_trainer_rules`**

| Column | Type | Notes |
|--------|------|-------|
| `id` | uuid PK | |
| `title` | text | Human-readable name |
| `triggers` | text[] | Match keywords / intents |
| `product` | text | Optional product tag (e.g. `PWR`) |
| `instruction` | text | What the AI must do / say / avoid |
| `priority` | enum | `advisory` \| `strong` \| `override` |
| `enabled` | boolean | Disable without deleting |
| `notes` | text | Internal-only context |
| `created_by` | uuid → profiles | |
| `created_at` / `updated_at` | timestamptz | |

**New enum:** `trainer_priority` (`advisory`, `strong`, `override`)

**RLS:** Admin/super_admin manage; agents read enabled rules only.

### Edge Function Integration

**File:** `supabase/functions/whatsapp-auto-reply/index.ts`

- New `loadTrainerRules()` — fetches enabled rules.
- New `matchTrainerRules(message, rules)` — keyword match against `triggers[]`.
- `generateAIAnswer()` now injects a prioritized `═══ TRAINER RULES ═══` block into the system prompt **above** the knowledge context.
- **Hard overrides** (`priority = override`) are framed as non-negotiable instructions that win over generic knowledge inference.
- Diagnostics now include `trainer_rules_matched`, `trainer_override_active`.

### Admin UI

**File:** `src/components/vanto/AITrainerPanel.tsx`
**Location:** Settings → AI Trainer (Sparkles icon)

Features:
- CRUD for trainer rules
- Priority selector (advisory / strong / override)
- Enable / disable toggle
- Trigger phrase manager (comma-separated, normalized)
- **Test Trainer sandbox** — type a sample inbound message and see which rules would match before saving

### Seeded Example: PWR Gender Routing

A hard-override rule is seeded:

| Field | Value |
|-------|-------|
| Title | PWR — Ask gender first |
| Triggers | `tired, fatigue, low energy, exhausted, energy` |
| Product | PWR |
| Priority | override |
| Instruction | Never say "PWR" alone. If user is male → recommend PWR Lemon. If female → PWR Apricot. If gender unknown → ask "Is this for a man or a woman?" before recommending. |

**Expected behavior** for inbound `"I'm always tired"`:
> *"For energy support, I'd first like to check — is this for a man or a woman?"*

After the user clarifies, the AI returns the correct gendered product.

## 2. Twilio / Maytapi Admin Privacy Lockdown

### Problem Solved
Regular agents could read the entire `integration_settings` table (which holds the admin's Twilio test recipient number, Maytapi credentials, webhook URLs, etc.) and could see the administrator's WhatsApp number in the Group Campaigns / Health panels. This leaked operator-only configuration to every authenticated user.

### Solution — Defense in Depth

**Layer 1 — Database RLS** (`integration_settings`)

The blanket "Authenticated users can view settings" policy was dropped and replaced with:

```sql
CREATE POLICY "Agents can view safe public settings"
ON public.integration_settings
FOR SELECT TO authenticated
USING (
  is_admin_or_super_admin()
  OR key IN ('auto_reply_mode', 'chrome_extension_heartbeat')
);
```

Only admin/super_admin can read sensitive keys. Agents see a tiny whitelist of UI-safe values.

**Layer 2 — Edge Function Role Check**

`supabase/functions/maytapi-health/index.ts` now:
- Requires a valid JWT
- Checks the caller's role via `user_roles`
- Strips `number`, `phone_id`, and other sensitive fields from the response for non-admins
- Gates the group listing endpoint to admin/super_admin only

**Layer 3 — UI Gating**

- `IntegrationsModule.tsx` — `TwilioHealthPanel` (which exposes webhook URLs and test recipient) is hidden behind an `isAdmin` check.
- `GroupCampaignsModule.tsx` — Health card no longer renders the administrator's WhatsApp number for non-privileged users; the API response is already redacted server-side.

## 6. Smoke Test Steps

### AI Trainer
1. Sign in as admin → **Settings → AI Trainer**
2. Verify the seeded PWR rule is visible
3. Open the **Test Trainer** box, type `"I'm always tired"` → confirm the PWR rule matches
4. Send a real WhatsApp inbound containing "tired" → reply should ask for gender first
5. Reply "I'm a woman" → AI should recommend **PWR Apricot**

### Privacy Lockdown
1. Sign in as a **non-admin agent**
2. **Settings → Integrations** → Twilio Health Panel must be **hidden**
3. **Group Campaigns** → admin's WhatsApp number must **not appear** in any health card
4. Open browser devtools → query `integration_settings` via the Supabase client → only `auto_reply_mode` and `chrome_extension_heartbeat` rows return
